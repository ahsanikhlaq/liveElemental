4. Methodology

This audit is a manual code-level review of the theme files listed in §3, evaluating each against the applicable WCAG 2.1 Level AA success criteria. Where a finding required dynamic verification (focus order, motion preference, screen-reader announcement), the relevant interaction was reproduced in a browser before being recorded as pass / fail / advisory.

  1. Manual code review. Each file was read line-by-line, with semantic HTML, ARIA attributes, keyboard handlers, focus management, motion handling, form labelling, and live regions traced from markup through to the JavaScript controllers in assets/dialog.js, assets/focus.js, and assets/cart-drawer.js.
  2. Colour-contrast calculation. Brand-palette ratios were calculated using the official WCAG sRGB → relative-luminance formula L = 0.2126R + 0.7152G + 0.0722B with the gamma-corrected channel transform.
  3. Locale audit. locales/en.default.json was checked to confirm every accessibility string key referenced by templates is present in the translations.
  4. Standards alignment. Findings were mapped to the W3C WCAG 2.1 Recommendation and cross-referenced against the WCAG 2.2 additions for forward-compatibility.

5. Issues Identified & Fixes Applied

Each issue below cites the file and line numbers in the audited commit (5aff11a). The failing source code and the corrected source code are both included so the fix can be independently verified.

FixedHighIssue 1 — Focus indicator removed on contact-form inputs

WCAG Criterion2.4.7 Focus Visible (Level AA)
Filesnippets/contact-form.liquid — lines 360–366

Failing code (pre-fix)

.contact-form__input:focus {
    outline: none;        /* removed indicator for ALL focus, including keyboard */
    border-color: #999;
}

Current code (post-fix), snippets/contact-form.liquid:360-366

.contact-form__input:focus {
  border-color: #999;
}

.contact-form__input:focus:not(:focus-visible) {
  outline: none;
}

Why this passes: the browser-default focus ring is suppressed only when focus came from a pointer device. Keyboard users see a clear focus ring on every contact-form field; mouse users see no change.

FixedHighIssue 2 — .fade-up ignored Reduce-Motion preference

WCAG Criterion2.3.3 Animation from Interactions (AAA), supports 2.2.2 Pause, Stop, Hide (A)
Fileassets/custom.css — lines 108–113

Current code (post-fix), assets/custom.css:97-113

.fade-up { /* translate / opacity entry transition */ }
.fade-up.is-visible { /* trigger state */ }

@media (prefers-reduced-motion: reduce) {
  .fade-up {
    opacity: 1;
    transform: none;
    transition: none;
  }
}

Why this passes: users with the Reduce-Motion OS preference now see static content with no slide-up transition, removing a vestibular trigger.

FixedHighIssue 3 — Scrolling promotion ignored Reduce-Motion preference

WCAG Criterion2.2.2 Pause, Stop, Hide (Level A)
Fileassets/scrolling-promotion.css — lines 209–212

Current code (post-fix), assets/scrolling-promotion.css:209-213

@media (prefers-reduced-motion: reduce) {
  .m-promotion--animated {
    animation-play-state: paused;
  }
}

An existing pointer-hover pause sits at assets/scrolling-promotion.css:42-45; combined with this query, the marquee is paused for any user expressing the OS preference and on hover for everyone else, satisfying SC 2.2.2 for content moving longer than five seconds.

FixedMediumIssue 4 — Contact-form messages not announced by screen readers

WCAG Criterion4.1.3 Status Messages (Level AA)
Filesnippets/contact-form.liquid — lines 202–230

Failing code (pre-fix)

<div class="contact-form__error" tabindex="-1" autofocus>...</div>
<div class="contact-form__success" tabindex="-1" autofocus>...</div>

Current code (post-fix), snippets/contact-form.liquid:202-230

<div class="contact-form__error"
     tabindex="-1" autofocus
     role="alert"
     aria-live="assertive"
     aria-atomic="true">...</div>

<div class="contact-form__success"
     tabindex="-1" autofocus
     role="status"
     aria-live="polite"
     aria-atomic="true">...</div>

Why this passes: errors are announced immediately (assertive); success messages are announced after current speech finishes (polite). aria-atomic="true" ensures the entire message is read.
