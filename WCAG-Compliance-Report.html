4. Issues Found and Fixes Applied

Each issue lists the failing WCAG criterion, the file/line, the failing code, the replacement code, and the user impact. Every snippet below can be opened in the repository and verified.

FixedHighIssue 1 — Focus indicator removed on contact-form inputs

WCAG Criterion2.4.7 Focus Visible (Level AA)
Filesnippets/contact-form.liquid — lines 360–366

Failing code (pre-fix)

.contact-form__input:focus {
    outline: none;        /* removed indicator for ALL focus, including keyboard */
    border-color: #999;
}

Current code (post-fix), snippets/contact-form.liquid:360-366

.contact-form__input:focus {
  border-color: #999;
}

.contact-form__input:focus:not(:focus-visible) {
  outline: none;
}

Why this passes: the browser-default focus ring is suppressed only when focus came from a pointer device. Keyboard users see a clear focus ring on every contact-form field; mouse users see no change.

FixedHighIssue 2 — .fade-up ignored Reduce-Motion preference

WCAG Criterion2.3.3 Animation from Interactions (AAA), supports 2.2.2 Pause, Stop, Hide (A)
Fileassets/custom.css — lines 108–113

Current code (post-fix), assets/custom.css:97-113

.fade-up { /* translate / opacity entry transition */ }
.fade-up.is-visible { /* trigger state */ }

@media (prefers-reduced-motion: reduce) {
  .fade-up {
    opacity: 1;
    transform: none;
    transition: none;
  }
}

Trigger context: the entry animation is started by the IntersectionObserver in layout/theme.liquid:74-94, which adds .is-visible when an element enters the viewport.

Why this passes: users who set "Reduce Motion" in their OS see static content with no slide-up transition, removing a vestibular trigger.

FixedHighIssue 3 — Scrolling promotion ignored Reduce-Motion preference

WCAG Criterion2.2.2 Pause, Stop, Hide (Level A)
Fileassets/scrolling-promotion.css — lines 209–212

Current code (post-fix), assets/scrolling-promotion.css:209-213

@media (prefers-reduced-motion: reduce) {
  .m-promotion--animated {
    animation-play-state: paused;
  }
}

An existing pointer-hover pause sits at assets/scrolling-promotion.css:42-45; combined with this query, the marquee is paused for any user expressing the OS preference and on hover for everyone else, satisfying SC 2.2.2 for content moving longer than five seconds.

FixedMediumIssue 4 — Contact-form messages not announced by screen readers

WCAG Criterion4.1.3 Status Messages (Level AA)
Filesnippets/contact-form.liquid — lines 202–230

Failing code (pre-fix)

<div class="contact-form__error" tabindex="-1" autofocus>...</div>
<div class="contact-form__success" tabindex="-1" autofocus>...</div>

Current code (post-fix), snippets/contact-form.liquid:202-230

<div class="contact-form__error"
     tabindex="-1" autofocus
     role="alert"
     aria-live="assertive"
     aria-atomic="true">
  ...error text...
</div>

<div class="contact-form__success"
     tabindex="-1" autofocus
     role="status"
     aria-live="polite"
     aria-atomic="true">
  ...success text...
</div>

Why this passes: errors are announced immediately (assertive); success messages are announced after current speech finishes (polite). aria-atomic="true" ensures the entire message is read by NVDA, JAWS, and VoiceOver.

8. Methodology

  1. Manual code review of every file in §3 against each WCAG 2.1 AA success criterion (24 testable AA + 25 testable A).
  2. Semantic / ARIA verification — every interactive component (dialog, button, input, live region) traced from markup through the JS controllers in assets/dialog.js, assets/focus.js, assets/cart-drawer.js.
  3. Colour-contrast calculation — sRGB → relative luminance per WCAG formula with gamma-corrected channel transform; ratios reported to 2 decimals.
  4. Locale check — locales/en.default.json verified to contain every accessibility key referenced by the templates (accessibility.skip_to_text, accessibility.cart, accessibility.cart_count, accessibility.quantity, accessibility.increase_quantity, accessibility.decrease_quantity, actions.close_dialog).

This is a code-level audit. A complete certification would additionally include automated scans (axe-core / Lighthouse) and a session with assistive-technology users.

9. Recommended Next Steps for the Client

  1. Image alt text in the Shopify admin. Code at snippets/image.liquid:29 correctly forwards image.alt to the rendered <img>. Ensure every product, collection, and section image has descriptive alt text. Decorative images may keep empty alt intentionally.
  2. Optional contrast improvements (Advisory 1) — three single-line CSS edits lift the score to 100% Level AA without altering the brand palette.
  3. Annual / pre-launch re-audit any time the design system, colour scheme, or major layout sections change.
  4. Automated scans in CI (optional) — integrate axe-core or pa11y against staging URLs to catch regressions.

10. Compliance Statement

Based on this audit:

  • The liveElemental theme meets 100% of applicable WCAG 2.1 Level A success criteria.
  • The theme meets 96% of applicable WCAG 2.1 Level AA success criteria after the four fixes documented in §4.
  • The remaining 4% relates to brand-colour contrast on small text (Advisory 1) — a content/design decision the client may resolve with three optional CSS edits.
  • The theme is built on an accessible Dawn foundation and includes correctly implemented skip navigation, ARIA landmarks, modal dialog semantics, focus trapping, live regions for asynchronous status, multilingual labelling, motion-preference handling, and visible keyboard focus.
